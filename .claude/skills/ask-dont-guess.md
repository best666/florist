# 不猜测、不替用户做决定

本技能是 AI 工作协议，非建议。以下规则在**每次任务执行时自动生效**。

---

## 一、信息不足时：停止猜测，转向收集

### 触发条件（满足任一即激活）

- 定位 bug 根因时缺少**运行时信息**（日志、报错堆栈、网络响应、用户操作步骤）
- 同一问题**已尝试 1 次修复但未解决**
- 对原因有 ≥2 种不同猜测，无法从代码静态分析中区分
- 涉及**外部系统行为**（服务器响应、浏览器 API、第三方库行为）

### 执行规则

**禁止**：凭猜测连续写出多轮修复代码。第 1 次尝试失败后，**禁止**第 2 次猜测性修复。

**必须**：向用户发送结构化的信息请求，格式如下：

```
定位这个问题，需要以下信息（按优先级）：

1. [最关键的一条] — 例如：浏览器 Console 中 [xxx] 开头的日志
2. [次关键的一条] — 例如：Network 面板中该请求的 Status Code 和 Response Body
3. [可选] — 例如：操作系统 / 浏览器版本
```

每条信息请求必须说明**为什么需要它**和**它会排除哪类可能性**。

---

## 二、多方案时：列出选项，不替用户选

### 触发条件

- 同一需求有 ≥2 种实现方式，各有明显权衡
- 选择会影响现有功能范围或用户体验
- 涉及引入/移除第三方依赖
- 改动范围不明确（是只改 A 还是连 B 一起改）

### 执行规则

**禁止**：在用户未表态的情况下选择方案并开始实施。

**必须**：使用 `AskUserQuestion` 工具，每条选项包含：

| 要素 | 示例 |
|---|---|
| 选项标签 | "方案 A: 前端 Canvas 压缩" |
| 一句话描述 | 在浏览器端用 Canvas 压缩后再上传 |
| 优缺点 | 优点：不依赖服务端；缺点：大图可能 OOM |
| 推荐标注 | （推荐）— 仅当你确实有充分理由时 |

---

## 三、改动影响不确定时：先确认范围

### 触发条件

- 修改的组件/函数被 ≥2 个其他文件引用
- 修改涉及共享类型（`packages/contracts/`）
- 修改涉及 API 接口签名变更
- 可能影响其他平台（H5 vs 小程序）

### 执行规则

执行前发送确认：

```
此改动会影响以下文件/模块：
- [文件 A] — [影响描述]
- [文件 B] — [影响描述]

确认只修改 [范围] 还是需要同步调整关联文件？
```

## 四、操作准则速查

| 场景 | ❌ 禁止 | ✅ 必须 |
|---|---|---|
| bug 定位，无日志 | 连写 3 种不同修复去"试试" | 第 1 次失败后立即向用户要日志/截图 |
| 多个技术方案 | 自己选一个开始写代码 | AskUserQuestion 列出选项 |
| 不确定改动范围 | 只改当前文件，让关联文件报错 | 搜索引用 → 列出影响 → 让用户确认 |
| 服务器行为不确定 | 猜测"应该是 xxx 导致的" | 让用户 curl 验证或提供 Network 截图 |
| 用户描述模糊 | 按自己理解直接动手 | 用 1-2 句话复述理解，确认后再动手 |

## 五、反面案例（来自本项目的实际教训）

**场景**：用户报告 `/api/image/upload` 返回 500。

**❌ 实际发生（连猜 5 轮全错）**：
```
第1轮：猜测 EXIF 方向导致 sharp 裁剪越界 → 修改了 rotate 逻辑 ✗
第2轮：猜测 body parser 100KB 限制 → 加了 json({ limit: '20mb' }) ✗
第3轮：猜测 res.data 类型不兼容 → 重写 parseResponse ✗
第4轮：猜测 CORS 导致响应体为空 → 改 URL 构建 ✗
第5轮：猜测 fail 回调被错误触发 → 加 complete/fail 兜底 ✗
// 其实只是 res.statusCode === 200 太严格，服务器返回 201
```

**✅ 应该做**：
```
第1轮修复（EXIF 方向确实有 bug）→ 修完 → 用户仍报 500
此时应立刻转向收集信息：
→ "请打开浏览器 Console，把 [uploadPreparedImage] 开头的日志复制给我"
→ 拿到 statusCode: 201 → 一行修复
```

**教训**：修复尝试 ≤1 次。如果第 1 次没解决，问题不在你已知的信息范围内，必须向用户收集新信息。

## 六、与项目其他技能的协作

本技能是底层的"工作协议"，与 `.claude/skills/debug.md` 互补：
- `debug` 描述的是**调试方法**（定位根因 → 最小修改 → 输出 diff）
- `ask-dont-guess` 描述的是**协作模式**（缺信息就问，多方案就列）

当 `debug` 流程中的"定位根因"步骤卡住时，本技能自动接管：停止分析，转向信息收集。
