# 批量修复协议

## 触发条件

修改涉及 ≥3 个文件，且文件间无强依赖。

## Phase 1: 确认范围

**禁止直接修改。** 先输出清单让用户确认：

```
此改动影响 N 个文件：

| 文件 | 内容 | 行数 |
|---|---|---|
| src/a.ts | [改什么] | 3 |
| ... | ... | ... |

确认批量修改？
```

## Phase 2: 流水线执行

按文件数分级：

| 规模 | 策略 |
|---|---|
| 3-4 个 | 主线程顺序改，最后一次 typecheck |
| 5-9 个 | 主线程分批改，每批完 typecheck |
| ≥10 个 | Agent 并行读 → 主线程改 → 校验 |

流水线：`读（Explore Agent 并行）→ 改（按依赖排序）→ 校验（typecheck）`

## Phase 3: 汇总报告

```
## 批量修复报告
**背景**: [为什么改]
**范围**: N 文件, M 处修改
**风险**: [影响面大的变更]

| 文件 | 变更 |
|---|---|
| ... | ... |

**建议回归验证**: [需手动测试的页面]
```

## 约束

- 不改动独有业务逻辑
- 不做"顺手重构"
- 每批改完立刻 typecheck
- 处理中发现 ≥3 处重复逻辑 → [[component-library]]
